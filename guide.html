B 데이터 흐름 대원칙

⭐ 가장 중요 · 반드시 먼저 이해할 것

채팅창(작업 대화창)이 최신 원본의 기준점이다. github은 보기·이동용 서버일 뿐, 데이터 저장소가 아니다. (github에서 최신본을 가져온다고 생각하지 말 것)

github에 올리는 것 / 안 올리는 것 (헷갈리지 말 것)

올리는 것: html만 (메인홈에서 보고 이동하는 용도)

안 올리는 것: 엑셀 마스터·데이터 원본·개인정보 → 대표가 드롭박스에 보관

따라서 "엑셀을 github에 올려 링크로 받기"는 안 한다. 엑셀 받기는 화면에서 즉석 변환으로 한다.

데이터는 두 곳에서 바뀝니다:

  • ① 채팅창에서 수정
  • ② 테마홈 화면에서 직접 입력·수정

②의 경우, 화면에서 파일을 다운로드해 사용자가 채팅창에 전달합니다. → 이렇게 채팅창을 항상 최신으로 동기화합니다. 따라서 테마홈은 화면의 데이터를 html·엑셀로 다운로드하는 기능이 필수입니다.

무엇을 채팅창에 줄지 (데이터가 어디 담겼느냐에 따라)

  • 데이터가 엑셀 마스터에 따로 있는 홈 → 데이터 바뀌면 엑셀을 준다
  • 데이터가 html 안에 담기는 홈(예: 케이터링) → 데이터 바뀌면 html을 준다
  • 화면·기능(디자인)을 바꿨을 땐 → html을 준다
  • 헷갈리면 둘 다 주면 가장 안전

C 상단 헤더 (위→아래 순서 고정)

(1줄) 왼쪽: [← 공공의주방 홈] + [작업 대화 ↗] (둘 다 알약형 버튼) 오른쪽: "ARCHIVEAT · ○○" (영문 영역명, 작은 회색 라벨) (2줄) 작은 라벨: "테마 홈 · ○○" (3줄) 제목: "○○ 홈" (Hahmlet 명조, 크게) (4줄) 날짜·버전: "2026.06.26 · v○" (DM Mono, 회색, 오늘 자동) (5줄) 설명: 아래 D 규칙대로 한 줄
  • 왼쪽 맨 위에 알약형 버튼 2개 나란히: [← 공공의주방 홈](메인홈 index.html으로) + [작업 대화 ↗](이 영역 작업 대화창으로, 새 탭)
  • [작업 대화 ↗] → 화면에서 바로 그 영역 대화창에 가서 수정 요청·파일 전달 가능
  • 우상단은 ARCHIVEAT · ○○(영문) 라벨로 통일. (네모박스 X)

D 5줄 설명 정의

  • 형식: "○○부터 ○○까지 관리합니다" — 이 영역이 무엇을 하는 곳인지 한 줄.
  • 예) 케이터링: "행사 등록부터 견적·메뉴·정산까지 관리합니다."
  • 예) 재무·세무: "매출·매입과 분기 신고자료, 결산을 관리합니다."

데이터 출처·신뢰성("재무제표 기준 검증" 등)이나 조작 안내는 여기 넣지 않습니다. → 출처는 카드 안에, 운영 원칙은 맨 아래 ※ 안내(F)로 분리.

E 원본 다운로드 구역 "📦 원본 파일"

목적: 매번 채팅창에 "파일 줘" 하지 않고, 테마홈 하단에서 버튼 한 번으로 최신 html·엑셀을 받아 드롭박스에 백업 보관하기 위함. (문제 생겼을 때 원본으로 기준을 다시 잡는 용도이기도 함)

버튼 2개 — 모든 테마홈 공통 (둘 다 넣는다)

📦 원본 파일 📄 이 화면 HTML 받기 ← 모든 홈 📊 엑셀 마스터 받기 ← 모든 홈 (선택) 💬 대화창에서 받기 ← 전체 일괄 요청용
  • 📄 이 화면 HTML 받기 — 현재 화면을 통째로 다운로드 (document 전체를 Blob으로, 자기 자신 저장). 파일명 ○○.html
  • 📊 엑셀 마스터 받기 — 화면에 심어둔 데이터를 그 자리에서 .xlsx로 만들어 다운로드(즉석 변환). github에 엑셀을 올려 링크로 받는 게 아님. 받은 건 드롭박스에 보관.
★★ 버튼 작동 원리 (반드시 이해할 것) ★★

html 다운로드 버튼은 "html 파일 안에 들어있는 데이터"만 줄 수 있다.

브라우저 보안상, html 혼자서 PC·드롭박스 등 외부 파일을 가져올 수 없다.

→ 엑셀 받기 버튼이 작동하려면, 그 데이터가 html 안에 심어져 있어야 한다. (B형도 화면 구울 때 데이터를 html에 심어두면 버튼으로 받힘)

★★ 화면 표시 ≠ 파일 내용 (개인정보 핵심) ★★

html은 봉투, 데이터는 그 안의 편지. 화면에 안 보이게 해도, 파일을 뜯으면 데이터가 다 보인다.

공개(public) github에선 html에 담긴 모든 게 노출된다. (화면에 안 찍어도 파일 안에 있으면 노출)

→ 개인정보(연락처·주소·이름)는 html에 아예 담지 않는다. (I-2 참조)

★★ 화면에서 직접 입력하는 홈(localStorage)의 필수 주의 ★★

localStorage = 사용자가 화면에서 입력한 데이터를 브라우저에 저장하는 방식 (케이터링 등).

다운로드 버튼은 반드시 localStorage의 입력 데이터까지 파일에 담아 내보내야 합니다.

❌ 흔한 실수: 버튼이 localStorage를 빠뜨리고 빈 껍데기만 다운로드. (실제로 이 문제로 빈 파일이 받아진 적 있음 → 반드시 방지)

✅ 올바른 동작: "기존 최신본 + 화면에서 새로 입력한 데이터"가 합쳐진 완전한 파일이 다운로드되어야 함.

F 맨 아래 ※ 운영 안내 (모든 테마홈 공통)

※ 홈은 보기·이동만 합니다.
  · 데이터를 고칠 땐 → 마스터(엑셀/대화창)에서
  · 볼 땐 → 이 홈에서
  · 모든 파일은 같은 폴더(github)에 두어야 링크가 작동합니다.

데이터 입력 방식 — 홈 성격에 따라 둘 중 선택 (우열 아님)

테마홈은 데이터를 다루는 방식이 두 가지로 나뉩니다. 새 테마홈을 만들 땐 "이 홈에서 데이터를 어떻게 바꾸는가?"를 먼저 정하고 그에 맞게 설계합니다. 어느 쪽이 더 낫고가 아니라, 그 홈에서 일하는 방식에 맞는 쪽을 고르는 거예요.

유형 A — html 마스터 (화면에서 직접 입력)
  • 예: 케이터링 (행사 등록, 상태 변경)
  • localStorage 사용 O (화면 입력을 담기 위해)
  • 데이터가 html 안에 담김 → 바뀌면 html을 채팅창에 줌
  • 보통 단일 파일 (html 하나에 데이터·화면·도구 다 들어감)
  • 필수: 다운로드 버튼이 localStorage 데이터까지 담아야 함 (E 참조)
유형 B — 엑셀 마스터 (채팅창에서 정리)
  • 예: 업무일지·재무세무 (채팅에서 엑셀 수정 후 재생성)
  • localStorage 사용 X (데이터는 엑셀에)
  • html은 엑셀에서 구워낸 읽기전용 표시본 → 바뀌면 엑셀을 줌 (디자인 변경 시 html도)
  • 보통 여러 파일 (html + 엑셀 + 생성기 등 분리)

판단 기준: "화면에서 직접 입력하느냐" → 잦으면 A(html), 채팅으로 정리하면 B(엑셀).

I 엑셀 백업 + 개인정보 원칙

I-1. 엑셀 백업 필수

★ 모든 테마홈은 하단 📦에서 html·엑셀을 둘 다 받을 수 있어야 한다

(대표가 드롭박스에 보관 + 필요시 채팅창 전달용)

원본이 html인 홈(유형 A)도 엑셀 백업을 별도로 챙긴다.

즉 "html 하나면 충분"이 아니라 → 원본 html + 백업 엑셀, 둘 다 챙긴다.

  • ★ 기술적으로 html 하나에 데이터가 다 들어있어 보여도, 백업 원칙을 우선한다. (엑셀 백업을 안 챙겨도 되는 것처럼 넘어가지 말 것)
  • 엑셀 백업을 못 받는 홈이 있으면 → 그냥 넘어가지 말고, 반드시 사용자에게 "이 홈은 엑셀 백업이 없습니다. 만들까요?" 확인한다.

I-2. 개인정보 처리 (개인정보 있는 홈 — 예: 클래스)

개인정보(연락처·주소·이름 등)는 html에 아예 담지 않는다.

이유: 화면에서 가려도 파일을 뜯으면 데이터가 남아있고, 공개 github에선 그게 노출된다. (E의 '화면 표시 ≠ 파일 내용' 참조)

처리: html·하단 버튼 = 개인정보 뺀 현황만(인원수·회차·정산). 완전한 원본 = 드롭박스에만 보관 (github·html 어디에도 안 올림).

완전판을 화면에서 받아야 하면 → 그때 github 비공개 전환 등 별도 의논. 기본은 "개인정보 빼고 안전하게".

각 홈 구체 설명

  • 케이터링 = catering.html 하나에 데이터·화면·도구가 다 들어있는 단일 파일. 화면에서 행사를 직접 등록·수정하므로 html이 원본. 바뀌면 html을 채팅에 줌. + 백업용 엑셀도 챙김(I 원칙).
  • 업무일지 = 엑셀 마스터가 원본. 생성기(gen_report.py)가 엑셀을 읽어 worklog.html을 굽는다. 화면 직접입력 없음. 바뀌면 엑셀을 줌. 작동 필수 5개(worklog.html·업무마스터.xlsx·생성기·참고 html 2개).
  • 재무·세무 = 엑셀 마스터(경영기획)가 원본. html은 읽기전용 보기창(대시보드). 바뀌면 엑셀(또는 새 원본자료)을 줌.
  • 클래스 = 엑셀 마스터(class-master)가 원본. 생성기(gen_class.py)가 굽는다. 바뀌면 엑셀을 줌. 필수 3개. ★ 개인정보(수강생 연락처·주소·이름)는 html에 안 담음 — 화면·하단 엑셀은 현황(인원수·회차·정산)만, 완전판은 드롭박스에만. (I-2)

J-2. 새 테마홈 구조 확인 질문 (새 홈 생기면 그 방에 복사해 던진다)

[이 테마홈 데이터 구조 확인 질문] 1. 이 홈의 "데이터 마스터(원본)"는 무엇인가요? → html 자체 / 별도 엑셀 / 구글시트 / 기타 (정확히) 2. 데이터를 주로 어디서 입력·수정하나요? → 화면에서 직접 / 채팅창에서 받아 반영 / 둘 다 3. localStorage(브라우저 저장)를 사용하나요? (예/아니오) 4. 데이터가 바뀌면 채팅창에 무엇을 줘야 최신이 전달되나요? → html / 엑셀 / 둘 다 (이유와 함께) 5. 이 홈에서 다운로드 가능한 파일은? 입력 데이터까지 담기나요? → 특히 "엑셀로 백업받기"가 가능한지 꼭 확인 6. 이 테마홈 관련 전체 파일을 빠짐없이 나열해주세요. (작동 필수 파일 / 백업 히스토리를 나눠서. 총 몇 개인지) 한 줄 요약: "이 홈은 [마스터]가 원본이고, 바뀌면 [html/엑셀]을 주면 된다. 필수 N개."

K 파일명 규칙 (영어 통일 · 백업 구분용)

  • 모든 파일은 영어로. (한글은 404·번역 깨짐 위험)
  • 형식: [영역영문]-[무엇]-[버전].확장자
  • 영역 이름을 앞에 붙여 백업 폴더에서 구분되게 한다.
  • 구분자는 하이픈(-). 띄어쓰기·점(·) 금지.
종류예시
html (보는 화면, 고정)worklog.html · finance.html · catering.html
마스터·데이터 엑셀finance-master-2026.xlsx · worklog-master-v118.xlsx

❌ 금지: master-2026.xlsx 처럼 영역명 없이 두면 어느 홈 건지 백업 폴더에서 구분 안 됨.

영역 영문명: 업무일지 worklog · 재무세무 finance · 쿠킹박스 cookingbox · 클래스 class · 케이터링 catering · 마케팅 marketing

※ 적용 순서: 새로 만드는 파일부터 적용. 기존 파일은 각 방 작업할 때 천천히 바꾼다 (한 번에 안 바꿔도 됨).

L 충돌 시 · 적용 전 확인

이 지침을 적용하기 전에, 각 테마홈의 기존 구조·기능과 맞지 않는 부분이 있는지 먼저 점검한다.

충돌하거나 이 홈에 맞지 않는 부분이 있으면 → 기계적으로 적용하지 말고, 먼저 사용자에게 알리고 대화한 뒤 작업한다.

특히 "엑셀 백업이 없는 홈"이면 그냥 넘어가지 말고 반드시 확인한다 (I 참조).

단, 아래 필수는 예외 없이 맞춘다: 영어 파일명 · 메인홈 복귀 링크 · 헤더 순서(C의 1~5줄).
